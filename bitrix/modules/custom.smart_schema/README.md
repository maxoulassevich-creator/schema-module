# Smart Schema Enterprise для 1C-Битрикс / Аспро Премьер

Модуль является переносом WordPress-плагина `Smart Schema Enterprise for WooCommerce & Elementor` на 1C-Битрикс с изменённой моделью работы: анализируются не все товары, а по одному реальному URL-примеру для каждого типового шаблона.

## Что перенесено из WordPress-плагина

- База предложений «что внести».
- Статусы: `pending`, `applied`, `skipped`, `rolled_back`.
- Журнал действий.
- Анализ реального front-end HTML.
- Извлечение существующего JSON-LD и аудит базовых ошибок.
- Ручное подтверждение перед выводом.
- Защита от молчаливого перезаписывания подтверждённых пунктов.
- Опциональный AI-анализ через OpenAI Responses API и SOCKS5-прокси.

## Что изменено под Bitrix

WordPress-плагин работал от сущностей WordPress/WooCommerce: `post`, `term`, `wp_head`, WP-Cron, `wp_options`, `$wpdb`. В Bitrix-версии единица работы — тип страницы:

- `product_detail` — детальная карточка товара.
- `product_category` — категория/раздел товаров.
- `blog_list` — список блога/новостей.
- `news_detail` — детальная страница новости/поста.
- `sitewide` — Organization/WebSite.

После подтверждения пунктов модуль выводит JSON-LD динамически через событие `main:OnEndBufferContent` перед `</head>`. Данные берутся из текущей HTML-страницы, а не из одного просканированного товара. Это предотвращает ошибку, когда schema одного товара попадает на все товары.


## Исправление 1.0.1

В этой сборке исправлена ошибка прямого перехода в админку, когда файл `/bitrix/admin/custom_smart_schema.php` искал страницу модуля в `/local/modules/custom.smart_schema`, а модуль установлен в `/bitrix/modules/custom.smart_schema`.

Теперь архив рассчитан на установку в путь `/bitrix/modules/custom.smart_schema` и дополнительно содержит исправленный файл `/bitrix/admin/custom_smart_schema.php`. Обёртка сначала ищет админ-страницу в `/bitrix/modules/custom.smart_schema/admin/custom_smart_schema.php`, затем — в `/local/modules/custom.smart_schema/admin/custom_smart_schema.php`.

Если модуль уже был установлен из предыдущей сборки:

1. Распакуйте архив в корень сайта с заменой файлов.
2. Убедитесь, что существуют файлы:
   - `/bitrix/modules/custom.smart_schema/admin/custom_smart_schema.php`
   - `/bitrix/modules/custom.smart_schema/admin/menu.php`
   - `/bitrix/admin/custom_smart_schema.php`
3. Откройте `/bitrix/admin/custom_smart_schema.php`.
4. Если пункт меню не появился сразу, очистите кеш административного меню Битрикс или перелогиньтесь в админке.


## Установка

1. Скопируйте папку `bitrix/modules/custom.smart_schema` в корень сайта.
2. В административной панели установите модуль: Marketplace → Установленные решения → Smart Schema Enterprise для 1C-Битрикс.
3. Откройте `/bitrix/admin/custom_smart_schema.php`.
4. Заполните 4 URL-примера:
   - реальный товар;
   - реальная категория товаров;
   - список блога/новостей;
   - реальная новость/пост.
5. Запустите «Проанализировать 4 типовые страницы».
6. Во вкладке «Что внести» проверьте найденные данные, аудит, пример JSON-LD и кандидаты файлов шаблона.
7. Нажмите «Внести» только для тех пунктов, которые нужно выводить.

## Важные ограничения

- Модуль не может определить фактические файлы шаблона Аспро Премьер без запуска на вашем Bitrix-сайте. В архиве WordPress-плагина нет файлов Bitrix/Аспро.
- Автоопределение шаблонов использует поиск кандидатов в `/local/templates` и `/bitrix/templates`; итог нужно проверить разработчиком на тестовом контуре.
- Модуль не придумывает цену, наличие, бренд, FAQ, видео и даты. Если данные не найдены в HTML текущей страницы, соответствующие свойства не выводятся.
- Если на странице уже есть такой же тип JSON-LD и включена настройка «Не выводить дубли», модуль пропускает этот тип.
- Для production сначала проверьте результат в валидаторах Google Rich Results Test, Schema.org Validator и Яндекс.Вебмастер.

## Почему выбран динамический вывод, а не запись в 10 000 товаров

У товаров в Bitrix/Аспро используется общий шаблон детальной страницы. Поэтому корректная архитектура — подтвердить типы разметки на уровне шаблона/типа страницы и строить JSON-LD из текущего HTML/данных страницы. Хранить отдельную schema для каждого товара нельзя: это дорого, сложно поддерживать и рискованно при изменении цены/остатков.

## Файлы модуля

- `install/index.php` — установка, таблицы, регистрация обработчика события.
- `admin/custom_smart_schema.php` — админ-интерфейс.
- `lib/HtmlAnalyzer.php` — загрузка/анализ HTML.
- `lib/Scanner.php` — анализ URL-примеров и создание предложений.
- `lib/SchemaBuilder.php` — генерация JSON-LD.
- `lib/TemplateDetector.php` — поиск кандидатов файлов шаблона.
- `lib/Output.php` — динамический вывод подтверждённой разметки.
- `lib/Auditor.php` — базовый аудит существующего JSON-LD.
- `lib/AiClient.php` — опциональный AI-анализ.

## Версия 1.0.2

После обновления до 1.0.2 запустите анализ заново. Модуль учитывает существующую microdata-разметку Аспро Премьер и не должен повторно предлагать `Product`, `BreadcrumbList`, `BlogPosting`, `ItemList` и `CollectionPage`, если аналогичная разметка уже есть в HTML страницы.

Если в таблице остались старые пункты со статусом `pending`, новый запуск анализа пометит устаревшие предложения как `Не вносить`.

## Обновление 1.0.3

Добавлены три рабочих сценария:

1. **Ручная проверка страниц**  
   В настройках модуля появилось поле «Ручная проверка страниц». Вставляйте по одному URL на строку. Такие предложения привязываются к конкретной странице, а не ко всем товарам/категориям/постам.

2. **Проверка после внесения**  
   После нажатия «Внести» модуль загружает URL-пример и проверяет, появился ли в HTML JSON-LD-блок именно от модуля. Результат показывается в карточке предложения и в журнале. Также есть отдельная кнопка «Проверить».

3. **Замена существующей разметки**  
   Кнопка «Заменить своей» включает режим замены: перед вставкой своего JSON-LD модуль удаляет из HTML-буфера старые JSON-LD-блоки такого же типа и отключает корневой microdata `itemscope/itemtype` такого же Schema.org-типа. Файлы шаблона при этом не изменяются.

## Обновление 1.0.4

Исправлена ситуация, когда журнал AI/аудита показывал проблему на странице категории, но во вкладке «Что внести» оставались только sitewide-пункты Organization/WebSite.

Что изменено:
- аудит теперь учитывает не только JSON-LD, но и microdata `itemscope/itemtype`;
- для категории товаров microdata `ProductCollection` создаёт предложение замены на JSON-LD `CollectionPage`;
- для категории товаров microdata `ItemList` создаёт предложение замены на компактный JSON-LD `ItemList` из реальных карточек товаров;
- старые предложения, которые ранее были автоматически помечены как `skipped`, снова возвращаются в `pending`, если повторный анализ показывает, что они актуальны;
- проверка после «Внести» теперь также смотрит, осталась ли старая microdata того же типа при режиме «Заменить своей».

После обновления до 1.0.4 обязательно очистите кеш Битрикса и запустите анализ заново.


## Обновление 1.0.7

Добавлена проверка карточек товаров по требованиям Яндекс Товаров к разметке Product.

Модуль теперь ищет существующую разметку Product/ProductGroup в JSON-LD и microdata и проверяет, есть ли обязательные поля: name, brand, image, offers, а внутри Offer/AggregateOffer — price или lowPrice, priceCurrency и availability.

Если существующая разметка товара есть, но неполная, во вкладке «Что внести» появится пункт «Замена: карточка товара Product». Для таких пунктов используйте действие «Заменить своей», чтобы старый Product-блок был убран из HTML-буфера и вместо него был выведен корректный JSON-LD модуля.

В настройках добавлены параметры:

- «Проверять карточки товаров по обязательным полям Яндекс Товаров»;
- «Fallback для brand товара».

Fallback для brand используется только если бренд, производитель или издательство не найдены в HTML/JSON-LD. По умолчанию подставляется название организации из настроек модуля.

## Версия 1.0.8 — классификация ручных URL

Модуль не считает каждую вручную добавленную страницу обычной WebPage. Он анализирует полный HTML, видимый текст, title/H1/meta description, URL, JSON-LD и microdata. По этим признакам выбирается наиболее точный тип страницы: карточка товара, категория, блог/статья, контакты, о компании, доставка, оплата, возврат/гарантия, юридическая страница, FAQ, отзывы, поиск, бренды/издательства, услуга, мероприятие, видео, корзина, оформление заказа, профиль или безопасный fallback WebPage.

Если точный тип не подтверждён данными страницы, модуль не создаёт узкоспециализированную разметку и оставляет безопасный WebPage.

## Версия 1.0.9 — FAQ на аккордеоне

Исправлена ошибка, из-за которой FAQ-страницы Аспро Премьер (например, `/help/faq/`) выводились как `WebPage`, а не `FAQPage`. Причина: вопросы и ответы на таких страницах свёрнуты в Bootstrap-аккордеон, а прежний парсер искал только пару «заголовок + следующий абзац». Пары не находились, `faq_pairs` оставался пустым, и модуль откатывался на безопасный `WebPage`.

Теперь модуль дополнительно разбирает аккордеон: триггер-вопрос с `data-toggle="collapse"` / `data-bs-toggle="collapse"` / `aria-controls` связывается по `id` со скрытым блоком-ответом. В пару попадают только элементы, чей текст оканчивается на «?» и у которых есть непустой ответ, поэтому навигационные якоря и обычные `collapse`-переключатели не создают ложных вопросов.

Если реальные пары вопрос–ответ не найдены (например, контент подгружается через AJAX), модуль по-прежнему не выдумывает `FAQPage` и оставляет безопасный `WebPage`.

После обновления очистите кеш Битрикса, запустите анализ заново и подтвердите пункт `FAQPage` для FAQ-страницы.

## Версия 1.0.10 — юридические страницы не считаются FAQ

Исправлена ситуация, когда юридические/служебные страницы с подзаголовками-вопросами ошибочно определялись как FAQ. Например, политика «Обработка персональных данных» (`/info/obrabotka-personalnykh-dannykh/`) содержит разделы «Кто оператор?», «Какие данные мы используем?» — это структура документа, а не пользовательский FAQ, но модуль присваивал ей тип `faq_page` и разметку `FAQPage`.

Причина: в `detectKind()` признак «на странице есть 2+ пары вопрос-ответ» срабатывал раньше проверки по URL. Теперь явные пути (privacy/oferta/usloviya/delivery/payment/refund и т.п.) имеют приоритет, а «контентный FAQ» остался резервным правилом только для обычных информационных страниц. Дополнительно `FAQPage` больше не навешивается автоматически на юридические и транзакционные типы страниц.

Настоящие FAQ-страницы (URL `/faq/` или заголовок «вопросы и ответы») по-прежнему получают `FAQPage`.
