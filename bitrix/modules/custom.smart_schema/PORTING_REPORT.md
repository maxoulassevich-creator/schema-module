# Отчёт по переносу WordPress-плагина в модуль 1C-Битрикс

## 1. Что было в исходном WordPress-плагине

Исходный архив содержит плагин `smart-schema-woo-elementor` версии `1.4.2-enterprise`.

Ключевые классы исходника:

- `SSWE_DB` — таблицы предложений и журнала.
- `SSWE_HTML_Analyzer` — загрузка реального URL и анализ HTML.
- `SSWE_Auditor` — аудит найденного JSON-LD.
- `SSWE_Schema_Builder` — генерация Product, ProductGroup, Offer, WebPage, BlogPosting, FAQPage, VideoObject, BreadcrumbList, CollectionPage, ItemList, Organization, WebSite, LocalBusiness.
- `SSWE_Scanner` — сканирование сайта, постов, товаров и терминов.
- `SSWE_Queue` — фоновая очередь на WP-Cron.
- `SSWE_Output` — вывод подтверждённой разметки через `wp_head`.
- `SSWE_Admin` — вкладки «Обзор», «Настройки», «Что внести», «Журнал».
- `SSWE_AI_Client` — OpenAI Responses API и SOCKS5-прокси.

## 2. Почему прямой перенос невозможен

WordPress-версия зависит от WordPress/WooCommerce API: `WP_Query`, `get_post`, `get_terms`, `wc_get_product`, `wp_head`, `wp_options`, `$wpdb`, WP-Cron и AJAX-хуки WordPress. В 1C-Битрикс этих сущностей нет.

Для Bitrix/Аспро правильная единица работы — не отдельный товар, а типовая страница/шаблон компонента. Поэтому модуль не сканирует 10 000 товаров. Он анализирует 4 URL-примера и затем выводит подтверждённую разметку динамически на страницах соответствующего типа.

## 3. Что реализовано в Bitrix-модуле

- Установка как стандартного модуля `/bitrix/modules/custom.smart_schema`.
- Таблицы:
  - `b_custom_smart_schema_proposals`;
  - `b_custom_smart_schema_logs`.
- Админ-страница `/bitrix/admin/custom_smart_schema.php`.
- Вкладки:
  - «Обзор»;
  - «Настройки»;
  - «Что внести»;
  - «Журнал».
- Анализ 4 типовых URL.
- Поиск кандидатов файлов шаблона в `/local/templates` и `/bitrix/templates`.
- Генерация JSON-LD для:
  - Product;
  - BreadcrumbList;
  - FAQPage;
  - VideoObject;
  - CollectionPage;
  - ItemList;
  - BlogPosting;
  - Organization;
  - WebSite.
- Ручное подтверждение перед выводом.
- Динамический вывод через `main:OnEndBufferContent`.
- Пропуск дублей, если такой тип JSON-LD уже есть на странице.
- Опциональный AI-анализ через OpenAI Responses API.

## 4. Как модуль решает вашу ключевую проблему

Требование: «не сканировать абсолютно всё и не вставлять микроразметку каждому товару отдельно».

Решение в модуле:

1. Администратор задаёт один реальный товар и другие URL-примеры.
2. Модуль анализирует HTML этих страниц.
3. Модуль формирует предложения «что внести» на уровне типа страницы.
4. После подтверждения модуль выводит schema для текущей страницы динамически.
5. Для каждого товара данные берутся из текущего HTML этого товара, а не из сохранённого JSON одного примера.

## 5. Что нужно проверить на тестовом Bitrix-сайте

- Соответствие определения типа страницы реальной структуре URL.
- Кандидаты файлов шаблона Аспро Премьер.
- Корректность извлечения цены, валюты, наличия, артикула и бренда из HTML карточки товара.
- Нет ли уже встроенной микроразметки Аспро/Битрикс, чтобы не получить дубли.
- Результаты в валидаторах Google, Schema.org и Яндекс.Вебмастер.

## 6. Что я не мог подтвердить без доступа к Bitrix-сайту

- Фактические пути файлов шаблонов Аспро Премьер на вашем проекте.
- Реальные ID инфоблоков каталога, блога и новостей.
- Реальную структуру HTML карточки товара, категории, блога и новости.
- Наличие или отсутствие встроенной микроразметки в текущей версии вашего установленного решения Аспро Премьер.

Модуль подготовлен так, чтобы эти данные определялись уже на вашем окружении после ввода URL-примеров.
